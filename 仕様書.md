# AviQtl 0.1.0 完全仕様書

バージョン: 0.1.0-pre
作成日: 2026-05-16
目標: プレビュー画面（Qt Quick 3D / QRhi 描画）が最低限動作し、タイムラインの操作と同期する状態

***

## 第1章 アーキテクチャの根本思想

AviQtlは「データ指向設計（DOD）」と「ECS（Entity Component System）」を中核とし、UI・正本データ・補間・描画の各責務を完全に分離した「Dual-State（二重状態）アーキテクチャ」を採用する。いかなるレイヤーも、自分の責務外のデータを所有・変更してはならない。

### グラフィクスバックエンドの選定根拠

旧実装ではFilamentおよびDiligent Engineを検討したが、いずれも Qt の内部レンダリングスタックとの統合に追加コストが発生する外部エンジンである。

**Qt Quick 3D + QRhi ネイティブ統合**を採用する理由:

- Qt 6 の QRhi（Qt Rendering Hardware Interface）は Qt Quick / Qt Quick 3D のレンダリング基盤そのものであり、外部エンジンを持ち込む必要がない
- `QQuickWindow::rhi()` で取得した `QRhi*` を描画レイヤーがそのまま利用するため、SwapChain 競合が構造的に発生しない
- シェーダーは Vulkan-style GLSL を一度記述し、`qsb` ツールが SPIR-V / MSL / HLSL Shader Model 5.0 へ自動変換するため、クロスプラットフォーム対応がビルドシステムで完結する
- `QRhiComputePipeline` / `QRhiCommandBuffer::beginComputePass()` により Compute Shader を Qt RHI の枠組みで直接扱える
- `QSGRenderNode` または `beforeRendering` + `beforeRenderPassRecording` シグナルにより、Qt Quick のシーングラフと同一コマンドバッファ・同一レンダーパスに描画コマンドを注入できる
- `Qt::GuiPrivate` へリンクし `#include <rhi/qrhi.h>` するだけで利用可能（Qt 6.6 以降は準公開 API として文書化済み）
- libstdc++ / libc++ 問題が存在しない（Qt のビルドに準じる）

## 第2章 レイヤー定義と責務

### Layer 1: QML（UI層）
- 責務: ユーザー入力の受付とBridgeへのコマンド発行。画面表示のみ。
- 禁止事項: データの正本を持つこと。補間の計算を行うこと。描画コマンドを発行すること。
- 接点: CoreBridge（QML_SINGLETON）のみを通じてC++と通信する。

### Layer 2: Bridge / DocumentModel（正本層）
- 責務: プロジェクトの全データ（クリップ・キーフレーム・エフェクト定義）を木構造で保持する唯一の正本。Undo/Redoスタック・シリアライズ（.aviqtlファイルへの保存/読込）・プラグイン定義のロードを担う。
- 禁止事項: 描画処理を行うこと。補間値を自ら計算すること。
- 接点: 構造変化（クリップの追加/削除/構成変更）時のみECS WorldへBakeをトリガーする。

### Layer 3: ECS World（計算キャッシュ層）
- 責務: DocumentModelからBakeされたエンティティと、毎フレームの補間・変換計算を担う。データはすべてSoA（Structure of Arrays）形式のPOD（Plain Old Data）フラットバッファとして保持する。
- 禁止事項: QMLと直接通信すること。DocumentModelのデータ構造を変更すること。
- System実行順序: CommandSystem → InterpolationSystem → TransformSystem → RenderSystem

### Layer 4: QRhi 描画エンジン層
- 責務: ECSのRenderSystemから渡された行列・パイプラインステート・テクスチャを、`QRhi` / `QRhiCommandBuffer` を通じて Vulkan / Metal / D3D12 で描画する。
- 禁止事項: データの加工・補間・ロジックの実行。`QRhi` インスタンスの所有（所有は Qt Quick シーングラフ側）。
- バックエンド選択:
  - Linux:   Vulkan（Qt がデフォルト選択、`QSG_RHI_BACKEND=vulkan` でも明示可）
  - macOS:   Metal（Qt がデフォルト選択）
  - Windows: D3D12（Qt がデフォルト選択、D3D11 フォールバック有）
- `QRhi` インスタンスの所有権は常に Qt Quick シーングラフが持つ。描画エンジン層は `window()->rhi()` による借用のみ行う。

## 第3章 Qt RHI 統合アーキテクチャ

### シグナル接続

Qt 6 の `beforeRendering` シグナルはフレーム先頭（`QRhi::beginFrame()` 呼び出し後、レンダーパス開始前）に発行される。**描画コマンドの記録はレンダーパス内でのみ可能**であるため、リソース更新（バッファ書き込み等）と描画コマンド記録を別シグナルで分離する必要がある。

**正しいシグナル接続:**

```
sceneGraphInitialized シグナル (render thread / 1度のみ)
└─ RhiRenderer::initialize()
    └─ window()->rhi() で QRhi* を取得（nullptrチェック必須）
       window()->rhi()->nativeHandles() で必要に応じてネイティブハンドルを参照
    └─ QRhiGraphicsPipeline / QRhiComputePipeline を事前生成
    └─ PSOCache::init() を呼び出す

beforeRendering シグナル (render thread / QRhi::beginFrame() 後・レンダーパス開始前)
└─ RhiRenderer::frameStart()
    └─ QRhi::nextResourceUpdateBatch() でバッチを取得
    └─ uniformバッファ / テクスチャアップロード等のリソース更新をエンキュー
    └─ Compute Pass が必要な場合: beginComputePass() → dispatch() → endComputePass()

beforeRenderPassRecording シグナル (render thread / メインレンダーパス記録中)
└─ RhiRenderer::mainPassRecordingStart()  ← アンダーレイとして描画する場合
    └─ ECS RenderSystem → cb->setGraphicsPipeline() / draw() 等を記録
    ※ Qt Quick UI が描画コンテンツの上にオーバーレイ合成される

afterRenderPassRecording シグナル (render thread / メインレンダーパス記録中)
└─ アンダーレイではなくオーバーレイとして描画する場合はこちら

sceneGraphInvalidated シグナル (render thread)
└─ RhiRenderer::cleanup()
    └─ 全 QRhiResource の releaseAndDestroyLater() / delete
```

すべての接続は `Qt::DirectConnection` で行う（render thread 必須）。

**注意**: `QQuickWindow::rhi()` が有効値を返すのは `sceneGraphInitialized` シグナル以降に限られる。`beforeRendering` 初回時点での利用は nullptrクラッシュのリスクがあるため、必ず `sceneGraphInitialized` 内で初期化を完結させること。

### Offscreen テクスチャ方式（推奨アーキテクチャ）

動画編集プレビューは「プレビュー領域をひとつの `QRhiTexture` にレンダリングし、QML の `Item` として表示する」**Offscreenテクスチャ方式**が最も堅牢である。Qt 6 では `QQuickRhiItem`（Qt 6.7 以降）または `QRhiTextureRenderTarget` を用いた自前オフスクリーンパスで実現する。

```
[ECS RenderSystem]
    │ 毎フレーム・render thread 上で実行
    ├─ beginComputePass() / dispatch() / endComputePass()  ← エフェクト適用
    └─ beginPass(offscreenRenderTarget) / draw() / endPass()  ← オフスクリーン描画
         ↓
[QRhiTexture (offscreen)]
    │ Qt Quick シーングラフにテクスチャとして提供
    └─ QQuickRhiItem::renderToTexture() または
       QSGSimpleTextureNode で QML Item に表示

[Qt Quick シーングラフ (メインパス)]
    └─ タイムライン・パネル・ツールバー等の 2D UI を上から合成
```

この方式により、`beforeRenderPassRecording` への接続（= Qt Quick のメインレンダーパスへの割り込み）が不要になり、Qt Quick の UI コンテンツとプレビュー映像の合成・変形・クリッピングが QML レベルで自由に行える。

### `QQuickRhiItem` の利用（Qt 6.7 以降）

Qt 6.7 で導入された `QQuickRhiItem` は、QRhi ベースのカスタムレンダリングを QML `Item` として公開するための公式 API である。

```cpp
class PreviewRhiItem : public QQuickRhiItem {
    Q_OBJECT
    QML_ELEMENT
public:
    QQuickRhiItemRenderer *createRenderer() override;
};

class PreviewRenderer : public QQuickRhiItemRenderer {
public:
    void initialize(QRhiCommandBuffer *cb) override; // リソース初期化
    void synchronize(QQuickRhiItem *item) override;  // GUI thread → render thread 同期
    void render(QRhiCommandBuffer *cb) override;     // 毎フレームの描画
};
```

- `initialize()` は `sceneGraphInitialized` 相当のタイミングで呼ばれる。`QQuickRhiItemRenderer::rhi()` で `QRhi*` を取得する。
- `render()` の引数 `QRhiCommandBuffer*` は Qt Quick が管理するコマンドバッファである。このバッファに対して `beginPass()` / `endPass()` を行い、`QQuickRhiItem` が管理するオフスクリーンテクスチャに描画する。
- `QQuickRhiItem` を使わない場合は、`beforeRendering` + `beforeRenderPassRecording` の 2 シグナル構成（"RHI Under QML" パターン）を用いる。

### macOS Metal での注意事項

Metal バックエンドでは、copy 系操作（バッファアップロード等）をレンダーパス内の `render()` から行うことはできない。これらは必ず `beforeRendering` / `initialize()` のタイミング（レンダーパス開始前）に行う。

### スレッドモデル

- `QRhiCommandBuffer` はスレッドアンセーフのため Qt render thread 上でのみ操作する。
- Compute Dispatch の完了は `QRhiCommandBuffer::endComputePass()` 後に `QRhi::finish()` を呼ぶことで CPU 側で待機できる。フレーム間レース回避には後述の同期管理を参照。
- 将来的に `QRhiCommandBuffer` の録画をワーカースレッドで行う Secondary Command Buffer パターンへの拡張を想定する。

## 第4章 Compute Shader の同期管理

### 問題の定義

`QRhiCommandBuffer::dispatch()` は GPU 上で非同期実行される。完了を待たずに次フレームのグラフィクスパスで同じテクスチャを読み込むと**書き込み中テクスチャへの読み込みレースが発生する**。

### 解決策: Compute Pass と Graphics Pass の順序化

QRhi ではレンダーパスの順序と `QRhi::finish()` を組み合わせて同期する。

```cpp
// フレーム内で Compute Pass → Graphics Pass の順に発行することで自動的に順序保証
// ※ 同一フレーム内では QRhi バックエンドが必要なバリアを自動挿入する

// 1. Compute Pass（エフェクト適用）
cb->beginComputePass(resourceUpdates);
cb->setComputePipeline(m_effectPipeline.get());
cb->setShaderResources(m_effectSrb.get());
cb->dispatch(dispatchX, dispatchY, 1);
cb->endComputePass();

// 2. Graphics Pass（Compute 結果を入力テクスチャとして読む）
cb->beginPass(m_offscreenTarget.get(), Qt::black, {1.0f, 0}, nullptr);
cb->setGraphicsPipeline(m_renderPipeline.get());
cb->setShaderResources(m_renderSrb.get()); // Compute 出力テクスチャを SRV として参照
cb->setViewport(QRhiViewport(0, 0, size.width(), size.height()));
cb->draw(6); // Quad
cb->endPass();
```

### フレーム間レースの回避

前フレームのコマンドバッファが GPU で実行中に次フレームの記録を開始する場合、`QRhi::FramesInFlight` （通常 2）フレーム先行制御が QRhi バックエンドによって自動管理される。アプリケーション側で GPU フェンスを手動管理する必要はない。

ただし `QRhi::finish()` を呼ぶと CPU が GPU の全コマンド完了を待機するため、パフォーマンスクリティカルなパスでは使用を最小限にすること。クリーンアップ時には必ず呼ぶこと。

### テクスチャのリソース状態遷移

QRhi は Vulkan のリソース状態遷移バリアを自動管理する。アプリケーション側で手動バリアを記述する必要はない（`QRhiTexture::UsageFlags` に `StorageImage` と `SampledTexture` を両方指定しておくこと）。

## 第6章 ECSコンポーネント定義

すべてのコンポーネントはPOD（trivially_copyable）でなければならない。ポインタ・std::string・QStringの混入を厳禁とする。

```cpp
// エンティティのアクティブ状態
struct ActiveComponent         { bool active; };

// 描画トランスフォーム (補間後の確定値)
struct TransformComponent      { float x, y, z, scaleX, scaleY, rotX, rotY, rotZ, opacity; };

// キーフレーム参照 (DocumentModel上のデータへのIDによる間接参照)
struct KeyframeRefComponent    { uint32_t clipId; uint32_t effectId; };

// 描画テクスチャへの参照
// textureId: TextureRegistry が管理する QRhiTexture のインデックス
// psoId:     PSOCache が管理する QRhiGraphicsPipeline のインデックス
struct RenderableComponent     { uint32_t textureId; uint32_t psoId; uint32_t layer; };

// フレームバッファ区切りマーカー
struct RenderBoundaryComponent { bool clearBelow; uint32_t layer; };

// グループTransform伝播マーカー
struct GroupTransformComponent { uint32_t layerCount; };

// グローバル変換行列 (TransformSystemが書き込む)
struct GlobalMatrixComponent   { float m[16]; };
```

## 第7章 Systemの実行順序と責務

### 1. CommandSystem（毎フレーム先頭）
- CoreBridgeのコマンドキューを消化し、ECSの状態（ActiveComponent等）を更新する。

### 2. InterpolationSystem（毎フレーム）
- KeyframeRefComponentを持つエンティティに対し、DocumentModelのキーフレーム配列と現在フレーム（t）を入力として補間エンジンを呼び出す。
- 結果をTransformComponentの各フィールドに書き込む。

### 3. TransformSystem（毎フレーム）
- GroupTransformComponentを持つエンティティから子レイヤーへTransformを伝播させる。
- TransformComponentの値からGlobalMatrixComponentを計算する（Z=0 等倍正投影）。

### 4. RenderSystem（毎フレーム / Qt render thread 上で実行）
- `QRhiResourceUpdateBatch` を取得し、ユニフォームバッファ・テクスチャ更新をエンキューする。
- Compute Pass を開始し（`beginComputePass`）、エフェクトチェーンを `dispatch()` で適用後、`endComputePass()` を呼ぶ。
- RenderBoundaryComponentを持つエンティティをスキャンしRenderGraph DAGを構築する（第8章参照）。
- RenderableComponent + GlobalMatrixComponentを持つエンティティを処理する:
  - PSOCacheからブレンドモードに対応するQRhiGraphicsPipelineを取得する。
  - GlobalMatrixComponentの行列をユニフォームバッファ（QRhiBuffer）に書き込む。
  - `cb->setGraphicsPipeline()` / `cb->setShaderResources()` / `cb->draw()` を発行する。
- フレーム末尾: TextureRegistryのLRU解放処理を実行する。

## 第8章 RenderGraph DAGの定義

### DAGノードの定義

```cpp
struct RenderNode {
    uint32_t            layerId;
    uint32_t            inputTextureId;   // 入力テクスチャ (TextureRegistry / QRhiTexture)
    uint32_t            outputTextureId;  // 出力テクスチャ (TextureRegistry / QRhiTexture)
    uint32_t            effectPsoIds[8];  // エフェクトチェーン (最大8段)
    uint8_t             effectCount;
    bool                isRoot;           // trueの場合オフスクリーンテクスチャへ最終出力
};
```

### DAG構築アルゴリズム

1. RenderBoundaryComponentを持つエンティティを `layer` 昇順でスキャンする。
2. 各Boundaryを1つのRenderNodeとして生成する。
3. 下位レイヤーのoutputTextureIdを上位レイヤーのinputTextureIdとして接続する。
4. 最上位ノードの `isRoot = true` とし、`PreviewRhiItem` が管理するオフスクリーンテクスチャ（`QQuickRhiItemRenderer::renderTarget()` が返す `QRhiRenderTarget`）への最終出力とする。

### 中間テクスチャの管理

- 中間テクスチャはTextureRegistryから払い出し、フレーム終了時に返却する。
- 返却はRenderSystem末尾で一括処理し、LRUキャッシュに戻す。途中での解放は禁止する。

## 第9章 PSOキャッシュとシェーダー管理

### PSOCache

AviUtl互換のブレンドモードは15種以上あり、`PSO_Overlay` はブレンドステートだけでは実現不可能でありフラグメントシェーダー内での輝度条件分岐が必要である。PSOCacheは以下の通り定義する。

```
PSOCache
├─ PSO_Normal       (通常合成: src_alpha / one_minus_src_alpha)
├─ PSO_Add          (加算合成: src_alpha / one)
├─ PSO_Sub          (減算合成: カスタムシェーダー)
├─ PSO_Multiply     (乗算合成: dst_color / zero)
├─ PSO_Screen       (スクリーン合成: one / one_minus_src_color, プリマルチ対応)
├─ PSO_Overlay      (オーバーレイ: 輝度条件分岐フラグメントシェーダー必須)
├─ PSO_HardLight    (ハードライト: オーバーレイの入出力反転)
├─ PSO_SoftLight    (ソフトライト: カスタムシェーダー)
├─ PSO_Dodge        (覆い焼き: カスタムシェーダー)
├─ PSO_Burn         (焼き込み: カスタムシェーダー)
├─ PSO_Darken       (比較暗: カスタムシェーダー)
├─ PSO_Lighten      (比較明: カスタムシェーダー)
├─ PSO_Difference   (差分: カスタムシェーダー)
├─ PSO_Exclusion    (除外: カスタムシェーダー)
└─ PSO_Dissolve     (ディゾルブ: 確率的アルファシェーダー)
```

すべてのPSOは `sceneGraphInitialized` / `initialize()` のタイミングで事前生成する。フレームループ中の生成は禁止する。

各 PSO の型は `QRhiGraphicsPipeline`（グラフィクス）または `QRhiComputePipeline`（エフェクト Compute）である。

### シェーダー言語と変換パイプライン

すべてのシェーダーを **Vulkan-style GLSL** で記述する。`qsb` ツール（Qt Shader Tools モジュール）がビルド時に各バックエンド向けへ自動変換し、バイナリには埋め込まずに **.qsb ファイルとして配布ディレクトリへ配置する。**

アプリケーションは起動時に `effects/` および `objects/` ディレクトリをスキャンし、QMLファイルと同名の `.qsb` ファイルを動的に読み込む。

```
Vulkan-style GLSL (.vert / .frag / .comp)
    │
    ├─ SPIR-V 1.0        → Vulkan バックエンド
    ├─ GLSL ES 100 / 120 → OpenGL バックエンド  
    ├─ HLSL SM 5.0       → Direct3D 11/12 バックエンド
    └─ MSL 1.2           → Metal バックエンド
```

CMake での統合（外部ファイル出力）:

```cmake
# qt6_add_shaders のターゲット指定を外し、OUTPUTS でビルドディレクトリへ直接出力する
add_custom_command(
    OUTPUT ${OUTPUT_QSB}
    COMMAND Qt6::qsb ${QSB_ARGS} -o ${OUTPUT_QSB} ${SOURCE_GLSL}
    DEPENDS ${SOURCE_GLSL}
)
```

エフェクトのシェーダーは `ui/qml/effects/` に `.comp`（Compute Shader）または `.frag` として配置し、QMLとペアで管理する。

#### Compute Shader のリソースバインディング（Vulkan-style GLSL）

```glsl
#version 430

layout(local_size_x = 8, local_size_y = 8) in;

layout(binding = 0) uniform sampler2D inputTex;
layout(binding = 1, rgba8) uniform writeonly image2D outputTex;
layout(binding = 2) uniform Params {
    /* エフェクト固有パラメータ */
} params;

void main() {
    ivec2 id = ivec2(gl_GlobalInvocationID.xy);
    // ...
}
```

- `binding` は SPIR-V スタイルのバインディング番号を使用する（`qsb` が各バックエンドのレジスター番号へ自動マッピングする）。
- Compute Shader では `QRhiShaderResourceBinding::imageLoad()` / `imageStore()` / `sampledTexture()` / `uniformBuffer()` を組み合わせて `QRhiShaderResourceBindings` を構築する。

#### グラフィクスシェーダー（Quad 描画）

```glsl
// quad.vert
#version 440
layout(location = 0) in vec2 position;
layout(location = 1) in vec2 texCoord;
layout(location = 0) out vec2 v_texCoord;
layout(binding = 0) uniform UniformBlock { mat4 mvp; float opacity; } u;
void main() {
    gl_Position = u.mvp * vec4(position, 0.0, 1.0);
    v_texCoord = texCoord;
}

// quad.frag
#version 440
layout(location = 0) in vec2 v_texCoord;
layout(location = 0) out vec4 fragColor;
layout(binding = 1) uniform sampler2D tex;
layout(binding = 0) uniform UniformBlock { mat4 mvp; float opacity; } u;
void main() {
    fragColor = texture(tex, v_texCoord) * u.opacity;
}
```

## 第10章 補間エンジン

- DocumentModelが保持するキーフレーム配列の「生データ」を入力として受け取る独立モジュール。
- 線形（linear）・ベジェ（bezier）はC++でハードコード実装。
- カスタム補間（custom）はJSONの `expression` フィールドの数式をエンジン起動時にASTへプリコンパイルし、評価時のパース処理を排除して毎フレームの評価コストをほぼゼロにする。
- ECSのコンポーネントはキーフレームの概念を持たない。補間後の「確定したスカラー値」のみを受け取る。

## 第11章 Bake戦略

### Full-Bakeモード（SystemSettings.bakeStrategy = "FullBake"）
- プロジェクトロード時に全クリップのECSエンティティを生成しメモリに常駐させる。
- シーク時はActiveComponentのフラグ切り替えのみ。プレビューシークが最速。
- 大規模プロジェクトではVRAM/RAM消費が増大する。ハイスペックPC向け。

### On-Demandモード（SystemSettings.bakeStrategy = "OnDemand"）
- 再生ヘッド周辺の前後 `SystemSettings.onDemandPrefetchFrames` フレームのクリップのみを動的にエンティティ化・破棄する。デフォルト値は30フレーム。
- メモリ消費を最小化。ローエンドPC・大規模プロジェクト向け。

### Bakeのトリガー条件
- クリップの追加・削除・レイヤー移動・エフェクトの着脱（DocumentModelの構造変化時のみ）
- フレームごとのシークではBakeしない。

## 第12章 エフェクト・オブジェクトのプラグイン定義

### JSONスキーマ（オブジェクト）

```json
{
    "id":         "video",
    "name":       "動画ファイル",
    "type":       "object",
    "generator":  "VideoGenerator",
    "params":     { "file": "", "speed": 1.0 },
    "ui":         { "controls": [...] }
}
```

### JSONスキーマ（エフェクト）

```json
{
    "id":         "border_blur",
    "name":       "縁ぼかし",
    "type":       "effect",
    "shader":     "border_blur.comp",
    "params":     { "size": 10, "blur": 5.0 },
    "ui":         { "controls": [...] }
}
```

- `generator` の実装はC++として本体に登録する。
- `shader` は Vulkan-style GLSL の Compute Shader ファイル名（`.comp`）を指定する。ビルド時に `qsb` が `.qsb` へ変換し、**QMLファイルと同じディレクトリに配置される。**
- `params` と `ui.controls` はリビルドなしでJSONやQMLの変更のみで拡張可能。

## 第13章 Video / Image / Text / TextureRegistry APIの責務

### TextureRegistry（スレッド境界の明示）

```
[非同期デコードスレッド (FFmpeg)]
    │  デコード済みYUVフレームをCPUステージングバッファに書き込む
    ↓  (ロックフリーキューで受け渡し)
[render thread (Qt)]
    │  TextureRegistry::upload() を呼び出す
    │  QRhiResourceUpdateBatch::uploadTexture() で GPU へ転送
    │  転送完了後 textureId を RenderableComponent に書き込む
    ↓
[GPU]
    └─ QRhiTexture として描画パイプラインに供給
```

- `TextureRegistry::upload()` はrender threadからのみ呼び出す。
- 非同期スレッドからの呼び出しは禁止する。スレッド間の受け渡しはロックフリーキューで行う。
- LRUキャッシュで不要なテクスチャを自動解放する。解放タイミングはフレーム末尾のみとする。
- テクスチャの解放は `QRhiResource::deleteLater()` を使い、render thread 側でフレーム末尾に実行する。

### Video API
- FFmpegによる非同期デコード
- VA-API（Linux） / VideoToolbox（macOS） / DXVA2（Windows）によるハードウェアアクセラレーション
- dmabuf（Linux）によるVRAMへのゼロコピー転送（`QRhiTexture::importDmabuf()` 対応状況に依存）
- TextureRegistry経由でQRhiTextureへ転送
- LRUキャッシュによるフレーム管理

### Image API
- stb_imageによるPNG/JPEG/WebP読み込み
- MipMap自動生成（`QRhiTexture::genMipmaps()` またはビルド時プリベイク）
- `TextureRegistry::upload()` → `QRhiResourceUpdateBatch::uploadTexture()` で登録

### Text API
- Qt QuickのTextアイテムをオフスクリーン `QQuickRenderControl` で描画
- 描画結果（QImage）を `QRhiResourceUpdateBatch::uploadTexture()` でGPUへ転送（`TextureRegistry::upload()` 経由）
- 将来的にFreeType/SDF実装へ切り替え可能（API境界は変えない）
